# Phase 2 要約：アーキテクチャの整理 (Layered Architecture)

Phase 2では、1つのファイルに固まっていたコードを役割ごとに分割し、メンテナンスしやすく変更に強い「レイヤードアーキテクチャ」へとリファクタリングしました。

## 学んだこと
- **責務の分離**: 「リクエストの受付」「業務ロジックの実行」「データの保存」といった異なる役割を、それぞれ別のクラスやファイルに分ける方法を学びました。
- **ディレクトリ構造の設計**: `src/` 配下に `controllers`, `services`, `repositories` といったディレクトリを作り、コードを整理しました。
- **疎結合な設計**: 各層が自分の役割に集中し、他の層の詳細（例えばDBの具体的な操作など）を知らなくても動くようにすることで、部品の入れ替えが容易な構造を構築しました。

## 初めて登場した用語・概念
- **Layered Architecture (レイヤードアーキテクチャ)**: アプリケーションを複数の階層（レイヤー）に分けて設計する手法です。
- **Controller (コントローラー)**: ユーザーからのリクエストを一番最初に受け取り、適切なレスポンスを返す「受付嬢」のような役割です。
- **Service (サービス)**: アプリケーションのメインとなる業務ルールや処理（ビジネスロジック）を記述する「職人」のような役割です。
- **Repository (リポジトリ)**: データベースとのデータの出し入れだけを担当する「倉庫番」のような役割です。
- **Refactoring (リファクタリング)**: プログラムの動作（外から見た振る舞い）は変えずに、内部の構造を整理して綺麗にすることです。
- **DTO (Data Transfer Object)**: 異なる層の間でデータを安全に受け渡すために使われる、データ保持専用のオブジェクトです。
